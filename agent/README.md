# RMM Agent

Удалённый агент для захвата экрана и управления.

## Требования

- **Агент (Windows):**
  - [FFmpeg](https://ffmpeg.org/download.html) в PATH
  - [Go 1.24+](https://go.dev/dl/) для сборки
  - MinGW-w64 (для сборки с CGO)

- **Сервер:**
  - Python 3.10+
  - `pip install -r requirements.txt`

- **Клиент:**
  - Современный браузер (Chrome, Firefox, Edge)

## Сборка

### Вариант 1: Кросс-компиляция в Linux

```bash
cd agent
CC=x86_64-w64-mingw32-gcc CGO_ENABLED=1 GOOS=windows GOARCH=amd64 go build -o agent.exe .
```

### Вариант 2: Нативная сборка в Windows

1. Установите [MinGW-w64](https://www.mingw-w64.org/) или через MSYS2:

   ```powershell
   # Установка MSYS2
   winget install msys2.msys2
   # В терминале MSYS2:
   pacman -S mingw-w64-x86_64-gcc
   ```

2. Сборка:
   ```powershell
   $env:CC = "gcc"
   $env:CGO_ENABLED = "1"
   go build -o agent.exe .
   ```

## Запуск

### 1. Сервер (сигналинг)

```bash
python main.py
# или
uvicorn main:app --host 0.0.0.0 --port 8000
```

Сервер слушает на `http://0.0.0.0:8000`

### 2. Агент

Отредактируйте `main.go` строку 29, укажите адрес сервера:

```go
const serverURL = "ws://192.168.88.127:8000/ws/agent/agent1"
```

Запуск:

```bash
./agent.exe
```

### 3. Клиент (браузер)

Откройте в браузере:

```
http://192.168.88.127:8000
```

## Использование

1. Запустите сервер
2. Запустите агент на удалённом ПК
3. Откройте `http://<server-ip>:8000` в браузере
4. Управление мышью и клавиатурой работает автоматически
5. Переключение раскладки: **Ctrl+Shift**

## Команды управления

| Действие   | Описание                                      |
| ---------- | --------------------------------------------- |
| Мышь       | Движение, клики (левая/средняя/правая кнопка) |
| Клавиатура | Все клавиши + русский ввод                    |
| Ctrl+Shift | Переключение EN/RU индикатора                 |

## Устранение проблем

**FFmpeg не найден:** Добавьте FFmpeg в PATH или положите рядом с agent.exe

**Нет видео:** Проверьтеfirewall - порт 8000 должен быть открыт

**Тормозит:** Уменьшите fps в `main.go` строка 276: `"-framerate", "30"`

### Резюме по каналам:

### WebSocket соединения (FastAPI сервер) - используются ТОЛЬКО для сигнализации WebRTC (обмен SDP/ICE). Через них НЕ передаются рабочие данные.

### Внутри WebRTC создаются ДВА DataChannel:

## для JSON сообщений: команды мыши/клавиатуры, запросы информации

controlChannel (
"control"
)

### для BLOB данных: скриншоты, файлы

binaryChannel (
"binary"
)

### Видеопоток передается через отдельный WebRTC видеотрек, не связанный с DataChannels.

### Создание каналов: Браузер создает DataChannels, Go-агент получает их через

pc.OnDataChannel()

### Отправка/получение:

JSON →
channel.send(JSON.stringify())
/
channel.SendText()

BLOB →
channel.send(ArrayBuffer)
/
channel.Send([]byte)

Система координат и обработка мыши в RMM проекте
1. Архитектура системы координат
В проекте используются четыре различные системы координат:

Браузерные координаты - координаты курсора в окне браузера (clientX, clientY)
Видео элемент координаты - координаты внутри HTML video элемента с учетом масштабирования
Удаленные экранные координаты - физические координаты экрана на удаленной машине
Координаты активного окна - относительные координаты внутри конкретного окна приложения
2. Преобразование координат от браузера к удаленному экрану
Проблема масштабирования видео:
В функции
code
Copy
mapPointerToRemote()
происходит сложное преобразование координат:

code
Copy
Браузер (clientX, clientY) → 
Прямоугольник видео элемента → 
Учет соотношения сторон (letterbox/pillarbox) → 
Удаленные экранные координаты (remoteX, remoteY)
Ключевая проблема: видео элемент в браузере использует
code
Copy
object-fit: contain
, что означает, что реальное изображение может быть меньше HTML элемента с черными полосами по краям.

Процесс преобразования:
Получаем размеры HTML video элемента:
code
Copy
video.getBoundingClientRect()
Вычисляем соотношение сторон удаленного экрана:
code
Copy
remoteScreenWidth / remoteScreenHeight
Сравниваем с соотношением сторон video элемента:
code
Copy
rect.width / rect.height
Определяем размер фактически отображаемого изображения и отступы
Преобразуем координаты мыши в удаленные экранные координаты
3. Определение координат для окна приложения
Двойное преобразование:
Для определения позиции курсора относительно окна приложения используется двухэтапная система:

Этап 1: Браузерные координаты → Удаленные экранные координаты (функция
code
Copy
mapPointerToRemote
)
Этап 2: Удаленные экранные координаты → Относительные координаты окна:
code
Copy
winX = mapped.x - currentWindow.x
winY = mapped.y - currentWindow.y
Информация об активном окне:
Получается через Windows API (
code
Copy
getForegroundWindow
,
code
Copy
getWindowRect
)
Обновляется каждые 200ms через JSON сообщения в
code
Copy
controlChannel
Содержит: позицию (x,y), размеры (width, height), заголовок окна
4. Причины ошибок при смене разрешения
Основные проблемы:
Рассинхронизация размеров экрана:

code
Copy
actualScreenWidth/Height
(Go) ≠
code
Copy
remoteScreenWidth/Height
(браузер)
FFmpeg может захватывать в одном разрешении, а браузер получать информацию о другом
Задержка обновления информации:

Информация об экране обновляется через
code
Copy
screen_info
сообщения
Между сменой разрешения и получением новых данных есть временное окно
В это время преобразование координат происходит с устаревшими данными
Кэширование на уровне браузера:

Браузер может кэшировать размеры видео потока
WebRTC соединение может продолжать передавать кадры в старом разрешении
Различные методы определения разрешения:

code
Copy
getPhysicalScreenSize()
- через Windows API
code
Copy
robotgo.GetScreenSize()
- через библиотеку RobotGo
code
Copy
detectResolution()
- через анализ вывода FFmpeg
Результаты могут отличаться, особенно при высоком DPI
Последовательность событий при смене разрешения:
Пользователь меняет разрешение экрана
Go-агент:
code
Copy
startScreenWatcher()
обнаруживает изменение через 3 секунды
Go-агент: отправляет сигнал перезапуска FFmpeg
Go-агент: отправляет обновленную
code
Copy
screen_info
в браузер
Браузер: получает новые размеры экрана и обновляет
code
Copy
remoteScreenWidth/Height
Проблема: между шагами 1-5 координаты мыши преобразуются неправильно
5. Почему обновление браузера нормализует координаты
При обновлении страницы браузера происходит:

Полная пересинхронизация: все переменные сбрасываются в начальное состояние
Повторный запрос информации: браузер заново запрашивает
code
Copy
screen_info
через
code
Copy
sendControl("request_screen_info")
Новое WebRTC соединение: создается с актуальными размерами экрана
Свежие метаданные видео:
code
Copy
video.loadedmetadata
срабатывает с корректными размерами
6. Особенности работы на краях окна
Определение краев окна:
Функция
code
Copy
getWindowEdgeCursor()
использует константу
code
Copy
EDGE_THRESHOLD = 10
пикселей для определения близости к краю окна.

Проблема масштабирования на краях:
При неправильном преобразовании координат, курсор может "промахиваться" мимо краев
Ошибка в 1-2 пикселя критична для определения возможности изменения размера окна
Cursor стили (
code
Copy
nw-resize
,
code
Copy
ew-resize
и т.д.) применяются некорректно
7. Рекомендации по улучшению
Уменьшить интервал проверки разрешения с 3 секунд до 1 секунды
Добавить буферизацию координат во время переходных периодов
Реализовать более агрессивную синхронизацию между браузером и агентом
Добавить валидацию соответствия размеров видео потока и экрана
Кэшировать последние корректные координаты для сглаживания переходов
Эта система сложна из-за необходимости синхронизации множественных источников информации о разрешении экрана и координатах, что делает её уязвимой к временным рассинхронизациям при изменении конфигурации дисплея.
